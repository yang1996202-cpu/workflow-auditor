# Workflow Review

> 工作流复盘与重复模式发现 Claude Code Skill

---

## 一个关于命名的小故事

这个项目刚建的时候叫 `workflow-auditor`。

"审计"这个词用的挺顺手——扫描工作历史、发现重复、评估价值、出候选清单，听起来像是在做审计。但用了一段时间之后，我越来越觉得这个名字不对劲。

**审计是别人审你的。复盘是你审自己的。**

这个 skill 做的事情，是帮你回顾自己最近 30 天的工作，找出那些你重复做了好几遍、但还没有打包成工具的流程。这不是审计，这是复盘。两个字，意思差了一截。

于是我把名字改成了 `workflow-review`。

---

## 改名这件事本身，就是这个 skill 的最好注脚

改名的过程是这样的：

1. 我打开 VS Code，让 AI 帮我把 `workflow-auditor` 改成 `workflow-review`
2. AI 先改了 SKILL.md 里的文案
3. 然后发现：VS Code 的 skill 校验要求文件夹名和 `name` 字段必须一致——文案改了，但目录名还是旧的，报错
4. 改完目录名，再把 GitHub 仓库也重命名
5. 推送，完成

整个过程大概花了 10 分钟，中间发现了一个新坑：**如果你改了 `name` 字段但没改目录名，Claude 和 VS Code 会同时把新名字和旧名字都识别出来，变成两个 skill 并存，互相打架。**

这个坑现在已经写进了我的 agent 记忆里，下次不会再踩了。

---

但更有意思的地方在这里：

**我用来发现这个命名问题的工具，就是这个 skill 本身的前身。**

有一次我跑了一遍 workflow-auditor，扫出来的候选清单里有一条：

> "你经常在对话开始前用自然语言描述要做什么，但这个步骤没有被结构化，每次都要重新组织语言。建议打包成模板。"

这条建议我确认了，然后打包了。打包完再回头看，这件事本身就是"发现重复 → 打包 → 消除摩擦"的完整循环。

一个帮你发现重复工作的工具，本身也在被它自己的逻辑迭代。

---

## 这是什么

`workflow-review` 是一个 Claude Code Skill，做一件事：

**扫描你最近 30 天的工作历史，找出值得打包成工具的重复性手动流程，出一份候选清单，由你确认后再创建。**

它不会自动替你创建任何东西。候选清单出来之后，你来决定哪些值得做，哪些跳过。

---

## 为什么不直接自动创建

因为"重复"不等于"值得打包"。

有些事情你重复做，是因为它本来就不该被自动化——比如每次写文章前的选题思考，这个过程的摩擦本身有价值。

有些事情你重复做，是因为还没找到合适的时机——现在打包可能早了，等场景更稳定再做更好。

所以这个 skill 的设计原则是：**只给候选清单，判断权在你手里。**

## 安装

```bash
# 克隆到你的 Claude Code skills 目录
git clone https://github.com/yang1996202-cpu/workflow-review ~/.claude/skills/workflow-review
```

重启 Claude Code 后，输入 `/workflow-review` 即可触发。

---

## 触发词

在对话里说以下任何一句话都会触发：

- "帮我分析一下最近有哪些重复工作流"
- "看看有什么可以打包成 skill"
- "回顾一下我最近的工作模式"
- "找出值得自动化的事情"
- "有什么重复做的事情"

---

## 数据来源优先级

1. `~/.claude/usage-data/facets/*.json`（官方 `/insights` 生成的语义层，最准）
2. `session-meta` 和 `report.html`
3. 降级：`projects/*.jsonl`，需要你手动确认才走这条路

如果没有 usage-data，skill 会先让你跑一遍官方 `/insights` 再继续。

---

## 核心约束

- 不会把完整会话记录读入上下文，只提取统计摘要
- 候选清单出来后必须暂停等你确认，才会创建任何东西
- 每次只创建 1-2 个高置信度、范围窄的最小资产
